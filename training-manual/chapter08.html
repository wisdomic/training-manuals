Deploy + Git + การทำงานเป็นทีม

ระยะเวลา: 3 ชั่วโมง | วันที่ 4

🎯 วัตถุประสงค์การเรียนรู้
  • ใช้ Clasp Push อัปโหลดโค้ดขึ้น Google Apps Script ได้
  • ใช้ Clasp Deploy เผยแพร่ Web App ได้
  • ใช้ Git เก็บประวัติโค้ดและทำงานเป็นทีมได้
  • ใช้ GitHub จัดการโปรเจกต์และสำรองข้อมูลได้

8.1 Clasp Push — อัปโหลดโค้ดขึ้น GAS

Clasp (Command Line Apps Script Projects) ช่วยให้เราเขียนโค้ด GAS บนเครื่องตัวเองแล้วอัปโหลดขึ้น Google ได้:

สร้างโปรเจกต์ใหม่
# สร้างโปรเจกต์ Web App ใหม่
clasp create --type webapp --title "ระบบลงทะเบียนฝึกอบรม"

# หรือเชื่อมกับโปรเจกต์ที่มีอยู่แล้ว
clasp clone <Script ID>
อัปโหลดไฟล์ขึ้น GAS
# Push ไฟล์ทั้งหมดขึ้น Script Editor
clasp push

# Push แล้วเปิด Script Editor ในเบราว์เซอร์
clasp push --watch   # (auto push เมื่อไฟล์เปลี่ยน)

# เปิด Script Editor
clasp open
⚠️ สำคัญ: clasp push จะอัปโหลดไฟล์ .gs และ .html ทุกไฟล์ในโฟลเดอร์ — ไฟล์ที่ไม่ต้องการให้ใส่ใน .claspignore
ไฟล์ .clasp.json

ไฟล์กำหนดค่าที่สร้างอัตโนมัติเมื่อ clasp create/clone:

{
  "scriptId": "1NgnBHE9EaRo5TjtO_32eIIVoM9-Mmce07_a1Q9NWOe7s3AxwpcWDbyoB",
  "rootDir": "./gas"
}
ไฟล์ .claspignore

ระบุไฟล์ที่ไม่ต้องการ push ขึ้น GAS:

# ไม่ push ขึ้น GAS
node_modules/**
.git/**
README.md
package.json
.env
*.md
test/**
📋 สรุปคำสั่ง Clasp Push
คำสั่งหน้าที่
clasp create --type webapp --title "ชื่อ"สร้างโปรเจกต์ใหม่
clasp clone <scriptId>ดึงโปรเจกต์ที่มีอยู่
clasp pushอัปโหลดไฟล์ทั้งหมด
clasp pullดึงโค้ดจาก Script Editor
clasp openเปิด Script Editor ในเบราว์เซอร์
clasp statusดูไฟล์ที่จะถูก push

8.2 Clasp Deploy — เผยแพร่ Web App

เมื่อ push โค้ดเสร็จแล้ว ต้อง deploy เพื่อให้ผู้อื่นเข้าใช้งานได้:

Deploy เวอร์ชันใหม่
# Deploy ครั้งแรก
clasp deploy --description "v1 — เปิดใช้งานครั้งแรก"

# Deploy เวอร์ชันถัดไป
clasp deploy --description "v2 — เพิ่มระบบค้นหา"

# ดูรายการ Deploy ทั้งหมด
clasp deployments

# ผลลัพธ์:
# - AKfycbz9C1KO... @1 - v1 — เปิดใช้งานครั้งแรก
# - AKfycbxYZ2AB... @2 - v2 — เพิ่มระบบค้นหา
ยกเลิก Deploy
# Undeploy ด้วย Deploy ID
clasp undeploy AKfycbz9C1KO...

# Undeploy ทั้งหมด
clasp undeploy --all
URL Pattern ของ Web App

Development URL (ทดสอบ, เปลี่ยนตามโค้ดล่าสุด):

https://script.google.com/macros/s/<Script ID>/dev

Production URL (ใช้งานจริง, เปลี่ยนเมื่อ deploy ใหม่):

https://script.google.com/macros/s/<Deploy ID>/exec
💡 เคล็ดลับ: ใช้ dev URL สำหรับทดสอบ — ไม่ต้อง deploy ใหม่ทุกครั้ง แต่ต้อง clasp push ก่อนทดสอบ
🔄 Workflow การ Deploy
  1. แก้ไขโค้ดบนเครื่อง (VS Code / Kiro)
  2. clasp push — อัปโหลดขึ้น GAS
  3. ทดสอบด้วย dev URL
  4. ถ้าโอเค → clasp deploy --description "v3"
  5. แจ้ง URL ใหม่ให้ผู้ใช้ (หรือใช้ URL เดิมถ้า deploy ทับ)

8.3 Git Add, Commit, Push

Git คือระบบจัดการเวอร์ชันโค้ด — เหมือน "ประวัติย้อนหลัง" ของโค้ดทุกบรรทัด:

เริ่มต้นใช้ Git
# สร้าง Git repository ใหม่ (ครั้งแรก)
git init

# หรือ Clone โปรเจกต์ที่มีอยู่บน GitHub
git clone https://github.com/username/project-name.git
บันทึกการเปลี่ยนแปลง (Commit)
# ดูสถานะไฟล์ที่เปลี่ยน
git status

# เพิ่มไฟล์ทั้งหมดที่เปลี่ยน
git add .

# หรือเพิ่มเฉพาะไฟล์ที่ต้องการ
git add Code.gs
git add index.html admin.html

# บันทึก commit พร้อมข้อความอธิบาย
git commit -m "เพิ่มระบบค้นหาผู้ลงทะเบียน"

# ส่งขึ้น GitHub
git push origin main
Best Practices สำหรับ Commit Messages
✅ ข้อความ commit ที่ดี
  • เพิ่มระบบค้นหาผู้ลงทะเบียน
  • แก้ bug dropdown หลักสูตรไม่แสดง
  • เพิ่ม validation เบอร์โทร 10 หลัก
  • ปรับ UI ตารางให้ responsive
  • เพิ่ม Email แจ้งเตือนลงทะเบียน
❌ ข้อความ commit ที่ไม่ดี
  • fix
  • update
  • แก้ไข
  • asdfsdf
  • test test test
📝 สูตร Commit Message: [ทำอะไร] + [กับอะไร] — ต้องอ่านแล้วเข้าใจว่าเปลี่ยนอะไรบ้าง

8.4 Git Pull + การทำงาน 2 เครื่อง

เมื่อทำงานหลายเครื่อง (เช่น คอมที่ทำงาน + โน้ตบุ๊คที่บ้าน) ต้อง pull โค้ดล่าสุดก่อนทำงานเสมอ:

ดึงโค้ดล่าสุด
# ดึงโค้ดล่าสุดจาก GitHub
git pull origin main

# ดูประวัติ commit ล่าสุด
git log --oneline -5
⚠️ เมื่อเกิด Conflict (ขัดแย้ง)

เกิดเมื่อ 2 เครื่องแก้ไฟล์เดียวกันแล้ว push ซ้อนกัน:

# Git จะแสดง conflict ในไฟล์:
<<<<<<< HEAD
var title = 'ระบบลงทะเบียน v2';
=======
var title = 'ระบบลงทะเบียนฝึกอบรม';
>>>>>>> origin/main

# วิธีแก้:
# 1. เลือกว่าจะเก็บบรรทัดไหน (ลบ marker ออก)
# 2. git add . && git commit -m "แก้ conflict"
# 3. git push origin main
Git Stash — เก็บงานชั่วคราว
# เก็บงานที่ยังไม่ commit (ชั่วคราว)
git stash

# ดึงงานที่เก็บกลับมา
git stash pop

# ใช้เมื่อ:
# - ต้อง pull แต่มีงานที่ยังไม่ commit
# - ต้องสลับ branch แต่ไม่อยาก commit งานที่ยังไม่เสร็จ
🔄 Workflow การทำงาน 2 เครื่อง
1. Pull

git pull origin main

2. Code

แก้ไขโค้ด

3. Commit

git add + commit

4. Push

git push origin main

ทำซ้ำทุกครั้งที่เปลี่ยนเครื่อง!

🚨 กฎเหล็ก: Pull ก่อน Push เสมอ! ถ้าลืม pull แล้ว push จะเกิด conflict

8.5 GitHub — จัดการโปรเจกต์

GitHub คือเว็บไซต์เก็บ Git repository บน Cloud — ใช้เป็นทั้ง backup และเครื่องมือทำงานเป็นทีม:

สร้าง Repository ใหม่
  1. ไปที่ github.com > กดปุ่ม "New repository"
  2. ตั้งชื่อ repository (เช่น training-registration)
  3. เลือก Private (ไม่เปิดสาธารณะ) หรือ Public
  4. ติ๊ก "Add a README file"
  5. เลือก .gitignore template: Node
  6. กด "Create repository"
เชื่อม Local กับ GitHub
# เพิ่ม remote (ครั้งแรก)
git remote add origin https://github.com/username/project-name.git

# Push ครั้งแรก
git push -u origin main

# ตรวจสอบ remote
git remote -v
README.md — หน้าแรกของโปรเจกต์
# 🏛️ ระบบลงทะเบียนฝึกอบรม

ระบบลงทะเบียนฝึกอบรมออนไลน์ สำหรับสำนักงานพัฒนาฝีมือแรงงานสุรินทร์

## เทคโนโลยี
- Google Apps Script (GAS)
- Google Sheets (ฐานข้อมูล)
- Bootstrap 5 (UI)

## การติดตั้ง
1. Clone: `git clone https://github.com/username/project.git`
2. ติดตั้ง clasp: `npm install -g @google/clasp`
3. Login: `clasp login`
4. Push: `clasp push`

🛡️ GitHub เป็น Backup
ทำไมต้องใช้ GitHub?
  • Backup อัตโนมัติ — โค้ดอยู่บน Cloud ปลอดภัย
  • ประวัติทุก commit — ย้อนกลับเวอร์ชันเก่าได้ทุกเมื่อ
  • ทำงานเป็นทีม — หลายคนแก้โค้ดพร้อมกันได้
  • ทำงานหลายเครื่อง — sync โค้ดระหว่างคอมพิวเตอร์
